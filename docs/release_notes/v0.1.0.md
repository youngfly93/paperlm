# paperlm v0.1.0

_First public release — 2026._

## TL;DR

A drop-in [MarkItDown](https://github.com/microsoft/markitdown) plugin
that replaces the built-in PDF converter with a layout-aware pipeline
powered by [Docling 2.90](https://github.com/docling-project/docling)
and [PaddleOCR](https://github.com/PaddlePaddle/PaddleOCR). One install,
no API change:

```bash
pip install paperlm[docling]
```

```python
from markitdown import MarkItDown
md = MarkItDown(enable_plugins=True)
print(md.convert("paper.pdf").markdown)
```

For batch or server ingestion, `paperlm-batch` provides JSONL input/output,
Markdown + JSON sidecar artifacts, reusable worker subprocesses, per-task
timeouts, and hard RSS limits:

```bash
paperlm-batch paper_a.pdf paper_b.pdf --output-jsonl results.jsonl --output-dir outputs/
```

## What's in the box

- **Docling 2.90** (MIT) as the default engine — layout analysis,
  reading-order repair, table extraction, figure detection.
- **PaddleOCR PP-OCRv5 mobile** (Apache-2.0) for scanned PDFs —
  strong Chinese + English recognition at ~2.7 GB peak RSS.
- **pdfminer.six** as an always-available degradation fallback.
- **Plugin via `priority=-1.0`** so it wins over the built-in
  `PdfConverter` without any API change.
- **Structured IR** available via `result.ir` (non-stable API) for
  downstream RAG / agentic pipelines — typed `BlockType`, per-block
  bounding boxes, reading order, attrs for level/inline/caption links.
- **JSON sidecars** via `result.paperlm_json` and
  `result.paperlm_chunks_jsonl` for RAG pipelines that need structured
  output rather than only Markdown.
- **`paperlm-batch` CLI** for batch/server ingestion using guarded
  long-lived worker subprocesses with per-task timeout and RSS limits.
- **Week 3 post-processing**: per-page median-height inline-vs-block
  formula detection, CJK-aware GFM table rendering with cross-page
  merging, caption ↔ figure/table pairing, two-column reading-order
  repair.

## Install

```bash
# From PyPI
pip install paperlm[docling]            # default — Docling engine
pip install paperlm[docling,ocr]        # + PaddleOCR for scanned
pip install paperlm[all]                # all safe-license extras

# From source
git clone https://github.com/youngfly93/paperlm
cd paperlm
pip install -e '.[docling,dev]'
```

## Usage

### Python

```python
from markitdown import MarkItDown

md = MarkItDown(enable_plugins=True)
result = md.convert("paper.pdf")

# Stable MarkItDown API
print(result.markdown)

# Non-stable paperlm extensions (Python only — not surfaced in CLI/MCP)
print(result.engine_used)       # "docling" / "paddleocr" / "pdfminer" / "failed"
print(len(result.ir.blocks))    # structured blocks for downstream consumption
print(result.ir.warnings)       # degradation trail
```

### Engine selection

```python
md = MarkItDown(enable_plugins=True, paperlm_engine="ocr")       # force OCR
md = MarkItDown(enable_plugins=True, paperlm_engine="docling")   # skip scanned-check
md = MarkItDown(enable_plugins=True, paperlm_engine="fallback")  # pdfminer only
md = MarkItDown(enable_plugins=True, paperlm_enable_ocr=False)   # disable OCR in auto
```

### CLI

```bash
markitdown --use-plugins paper.pdf -o paper.md
markitdown --list-plugins        # must show "paperlm"
```

|     | TARGET: |     | Benchmarking |     |     | Table | Retrieval | for Generative |     | Tasks |

## Measured quality (8-fixture integration sweep)

Each fixture converted in an isolated subprocess on macOS-CPU. Full
data in [`benchmarks/phase4_integration.md`](../../benchmarks/phase4_integration.md).

| Fixture | Pages | Engine | Time | Peak RSS | Blocks |
|---|---|---|---|---|---|
| EN double-column bioRxiv | 27 | docling | 92 s | 2.3 GB | 441 |
| ZH double-column review | 10 | docling | 11 s | 1.4 GB | 183 |
| LLM-in-bio survey | 14 | docling | 17 s | 1.8 GB | 223 |
| Table-heavy benchmark | 12 | docling | 14 s | 1.4 GB | 190 |
| DeepSeek-R1 math-heavy | 86 | docling | 335 s | 3.0 GB | 903 |
| Graph-DB long survey | 60 | docling | 60 s | 2.1 GB | 457 |
| Synthetic scanned 5p | 5 | paddleocr | 95 s | 3.2 GB | 392 |
| Synthetic scanned 1p | 1 | paddleocr | 16 s | 2.7 GB | 45 |

**Every fixture under 4 GB peak RSS. Zero errors, zero warnings across
all 8.** Page-normalized throughput is 0.75–1.2 s/page for Docling on
non-math PDFs, 19 s/page for PaddleOCR.

Note: the 10.6 GB → 2.7 GB RSS reduction on scanned PDFs came from
switching PP-OCRv5 server → mobile models; quality is identical on our
Chinese bioinformatics fixture (same 45 text lines recognized). Details
in [`benchmarks/phase4_rss_probe.md`](../../benchmarks/phase4_rss_probe.md).

Phase 5 smoke profile, 3 fixtures × 4 tools on macOS CPU, isolated
subprocesses, 360 s per-task timeout:

| Tool | Successful | Median time | Scanned PDF | Title first line | License |
|---|---|---|---|---|---|
| MarkItDown baseline | 2/3 | 1.04 s | 0/1 ❌ | 0/3 | MIT |
| **paperlm** | **3/3** ✅ | 19.97 s | **1/1** ✅ | **2/3** ⭐ | Apache-2.0 |
| Docling standalone | 3/3 | 10.22 s | 1/1 | 1/3 | MIT |
| Marker CLI | **1/3** ❌ | 75 s | timeout | — | GPL-3 + OpenRAIL-M |

Two of three Marker runs exceeded the 360 s timeout on CPU

## Known limits (v0.1.0)

- **Bilingual layouts**: pages that place Chinese and English titles /
  abstracts side-by-side (common in some Chinese journals) can surface
  interleaved ordering. Single-language papers are clean.
- **Formula LaTeX extraction is opt-in** (`paperlm_enable_formula=True`):
  the CodeFormulaV2 VLM adds ~500 MB of model weight and noticeable
  latency. Without it, formulas render as `[formula]` placeholders
  that preserve layout but not content.
- **Figures** are rendered as `![](figure)` placeholders. Image export
  to disk is planned for v0.2.
- **Docling `document_timeout` is advisory** — very long math-dense
  documents (e.g. the 86-page DeepSeek-R1 fixture) can exceed the
  timeout and continue. Use engine-level budgets in your caller if you
  need a hard wall clock.
- **Docling cold-start is still expensive** on CPU. Use `paperlm-batch`
  or `DoclingWorkerPool` for batch/server workloads so the heavy
  converter is loaded in reusable guarded worker subprocesses.

## How reviewers can verify

The fast test suite deliberately **skips** every path that needs an
ML model or a missing core dep. That is correct but not conclusive.
To actually exercise the Docling / OCR / fallback paths:

```bash
python tests/fixtures/fetch.py                # build the corpus
pip install -e '.[docling,ocr,dev]'
make test-all                                 # ~3-4 minutes

# Or trigger the CI integration job on a fork:
gh workflow run test.yml --field ref=<branch>
```

## Full changelog

This is the first public release. The work happened in 5 weeks:

- **Week 1**: Skeleton, Docling adapter, IR, Markdown serializer, Week 1
  test corpus (3 fixtures).
- **Week 2**: pdfminer fallback, `EngineRouter` with degradation chain,
  scanned-PDF detection, PaddleOCR adapter, `[ocr]` extra.
- **Week 3**: Formula inline/block detection, CJK-aware GFM tables
  with cross-page merging, caption↔figure/table pairing, two-column
  reading-order repair.
- **Week 4**: PaddleOCR RSS fix (10.6 GB → 2.7 GB), GitHub Actions CI
  with 4 jobs, fixture corpus expanded to 8 PDFs with `fetch.py`,
  integration sweep, reviewer-audit fixes (deps-missing tests now skip
  gracefully, version consistency, Python bytecode kept out of source
  tree).
- **Week 5**: Release packaging, JSON sidecars, guarded subprocess
  benchmarks, `DoclingWorkerPool`, `paperlm-batch`, `docs/RELEASE.md`
  playbook, bilingual launch blogs, release notes (this file).
